# Backend Architecture & Features (NerdsOnCall Project)

This document breaks down the fundamental architecture of key backend features in your application: Payment Processing, Job Scheduling, and Role-Based Access Control (RBAC).

## 1. Payment Controller & Subscription Flow
Your application handles payments using a 3rd-party gateway (Razorpay) and securely provisions user subscriptions.

**The Pipeline:**
1. **Initiation:** The client requests a checkout, generating an `orderId` via Razorpay.
2. **Verification (`PaymentController`):** Once the user pays, the client sends the `orderId`, `paymentId`, and a cryptographic `signature` to your `/payment/verify` endpoint.
3. **Validation & Provisioning:** The backend verifies the signature. If valid, it locates the pending `Subscription`, marks it as `ACTIVE`, and sets the `startDate` and `endDate` based on the plan type (Basic = 1 month, Standard = 3 months, Premium = 1 year).
4. **Post-Processing:** The backend asynchronously triggers the `EmailService` to send a subscription receipt with a dynamically generated PDF attachment to the user.

## 2. Cron Schedulers (Background Jobs)
To handle time-based tasks without user interaction, your backend utilizes Spring Boot's `@Scheduled` annotation to run automated background jobs (Cron jobs) in the `SubscriptionSchedulerService`.

**Key Schedulers in your system:**
- **Daily Quota Reset (`0 0 0 * * ?`):** Runs exactly at midnight (12:00 AM) every day. It iterates through all active subscriptions and resets their `sessionsUsed` back to `0`.
- **Expiration Processing (`0 0 * * * ?`):** Runs every hour to check if any active subscriptions have passed their `endDate` and updates their status to `EXPIRED`.
- **Database Cleanup (`0 0 2 * * ?`):** Runs at 2:00 AM daily to delete `EXPIRED` subscriptions that are older than 30 days, preventing database bloat.
- **Analytics Logging (`0 0 1 * * ?`):** Runs at 1:00 AM daily to aggregate and log system health metrics (active vs. expired subscriptions, session usage percentages).

## 3. RBAC (Role-Based Access Control) & Security
Your application restricts access to endpoints based on the authenticated user's assigned role (e.g., Student, Tutor, Admin).

**The Pipeline:**
1. **Stateless Authentication:** You use JWTs (JSON Web Tokens). When a user logs in, the backend signs a token containing their User ID and Role, which is returned to the client.
2. **Request Filtering (`JwtAuthenticationFilter`):** On every subsequent request, a filter intercepts the API call, extracts the JWT from the `Authorization` header, validates its signature, and populates the Spring `SecurityContext`.
3. **Endpoint Security (`SecurityConfig`):** Broad rules are set up (e.g., `/api/auth/**` is public, while `/subscriptions/**` requires authentication).
4. **Method-Level Security:** You utilize `@EnableMethodSecurity(prePostEnabled = true)`, allowing specific controller methods to be guarded by annotations like `@PreAuthorize("hasRole('TUTOR')")`, ensuring a student cannot access tutor-only endpoints.

## Technical Interview Questions & Answers

### Payment Processing
**Q1: How do you ensure that a malicious user doesn't fake a payment success request to your backend to get a free subscription?**
**Answer:** "When our frontend sends the payment success data to the `PaymentController`, it includes an `orderId`, `paymentId`, and most importantly, a cryptographic `signature` generated by Razorpay. Our backend takes the `orderId` and `paymentId`, hashes them using our secret Razorpay key (which the frontend never sees), and compares it to the provided `signature`. If they don't match, the request is rejected as fraudulent."

**Q2: What happens if the server crashes right after verifying the payment, but before the subscription is activated or the email is sent?**
**Answer:** "This is a classic distributed systems problem. To handle this, we rely on Webhooks. Even if our frontend fails to notify our backend, or our server crashes during the synchronous `/payment/verify` call, Razorpay will fire an asynchronous webhook (e.g., `payment.captured`) to our server. We have a dedicated webhook listener that is idempotent—it checks if the subscription is already active, and if not, processes the activation. This ensures eventual consistency."

### Schedulers / Background Jobs
**Q3: What is a Cron Scheduler, and why did you use it instead of just running a `while(true)` loop with a thread sleep?**
**Answer:** "A Cron Scheduler is a time-based job scheduler. We use it via Spring's `@Scheduled` annotation to run maintenance tasks like resetting daily session quotas at midnight or cleaning up old database records at 2:00 AM. Using a `while(true)` loop with `Thread.sleep()` is bad practice—it ties up application threads, is prone to crashing, doesn't account for server restarts or daylight saving time changes, and makes it impossible to define complex schedules (like 'every day at midnight')."

**Q4: In your `SubscriptionSchedulerService`, what happens if your daily midnight quota reset fails halfway through?**
**Answer:** "To make the scheduler resilient, the logic should be wrapped in a database transaction (`@Transactional`). This way, if it fails halfway through updating user quotas, the transaction rolls back, and no users are left in an inconsistent state. We also wrap individual processing blocks in `try-catch` blocks and log errors, so a failure on one specific user's subscription doesn't crash the entire scheduled job."

### Role-Based Access Control (RBAC)
**Q5: Explain how Role-Based Access Control (RBAC) works in your application.**
**Answer:** "RBAC ensures users only access resources they are authorized for. In our app, when a user authenticates, their assigned role (e.g., TUTOR, STUDENT) is embedded into their signed JWT. On subsequent requests, our `JwtAuthenticationFilter` validates the token and extracts the role. We then use Spring Security's method-level security (`@PreAuthorize`) to restrict specific controller routes—for example, ensuring only users with the TUTOR role can create tutoring sessions."

**Q6: Why do you store the user's role in the JWT instead of querying the database on every request?**
**Answer:** "Our authentication mechanism is designed to be stateless. By embedding the role directly inside the JWT payload, the backend can determine the user's identity and permissions simply by verifying the token's cryptographic signature. This completely eliminates the need to perform a database query on every single API request just to check permissions, drastically improving application performance and reducing database load."
